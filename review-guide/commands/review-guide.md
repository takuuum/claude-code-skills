description: Generate a human-friendly review guide for code changes. Explains what was changed and provides a recommended reading order. Invoke when the user wants to understand changes before reviewing, or needs a reading guide for a PR/diff.

# Review Guide Generator

コードレビューを行う人間向けに「どのような変更か」「どのコードから読むべきか」をガイドします。

## Arguments

`$ARGUMENTS` には以下を指定できます（省略可）:
- `--pr <number>` / `-p <number>`: 指定した PR の変更をガイド対象にする
- `--diff` / `-d`: ステージング済みの差分をガイド対象にする
- `--base <branch>`: 比較ベースブランチを指定（デフォルト: main or master）

引数なしの場合は、ステージング済みの差分 → なければ `HEAD~1..HEAD` の差分を対象にします。

---

## Step 1: 変更の取得

`$ARGUMENTS` を解析して、以下のいずれかで変更を取得します。

- `--pr <number>` → `gh pr diff <number>` と `gh pr view <number> --json title,body,commits`
- `--diff` または引数なし → `git diff --staged`（空なら `git diff HEAD~1 HEAD`）
- `--base <branch>` → `git diff <branch>...HEAD`

以下も並列で取得します:
- `git log --oneline -10` — 直近のコミット履歴
- `git diff --stat HEAD~1 HEAD` または `gh pr view <number> --json files` — 変更ファイル一覧
- プロジェクトのルートにある設定ファイル（`package.json` / `go.mod` / `Cargo.toml` / `pyproject.toml` / `pom.xml` など）でフレームワーク・言語を特定

---

## Step 2: 変更の分類と概要分析

取得した差分とコミット履歴を分析して、以下を判定します。

### 変更カテゴリの判定

以下のカテゴリを検出します（複数該当可）:

| カテゴリ | 判定基準 |
|---------|---------|
| **新機能追加** | 新しいファイル・新しいエクスポート関数・新しい API エンドポイントの追加 |
| **バグ修正** | 条件分岐の修正・例外処理の追加・既存ロジックの訂正 |
| **リファクタリング** | 機能変更なし・名前変更・ファイル分割・抽象化 |
| **テスト追加/修正** | `*.test.*` / `*_test.*` / `*spec*` ファイルの変更 |
| **設定・インフラ変更** | CI/CD・Docker・env・設定ファイルの変更 |
| **依存関係の更新** | `package.json` / `go.sum` / `requirements.txt` 等の変更 |
| **ドキュメント** | `*.md` / `*.rst` / コメントのみの変更 |
| **破壊的変更** | 既存の公開 API・インターフェース・型定義の変更 |

### 変更規模の判定

- **Small**: 変更行数 < 50、変更ファイル数 ≤ 3
- **Medium**: 変更行数 < 300、変更ファイル数 ≤ 10
- **Large**: 変更行数 ≥ 300 または変更ファイル数 > 10

---

## Step 3: テスト仕様ガイドの生成

変更差分にテストファイル（`*.test.*` / `*.db-test.*` / `*.dbcl-test.*` / `*.e2e-test.*` / `*.e2ecl-test.*` / `*_test.*` / `*spec*`）が含まれる場合、テストコードを読み込んで仕様ガイドを生成します。

### テストファイルの読み込み

変更されたテストファイルの **全文** を Read ツールで読み込みます（差分だけでなくファイル全体）。これにより `describe` / `it` のネスト構造やセットアップコードを正確に把握します。

### 分析する観点

各テストファイルについて以下を抽出します:

1. **対象の振る舞い（What）**: `describe` / `it` / `test` のラベルから、テスト対象が「何をするか」を抽出
2. **入出力の仕様（Given-When-Then）**: Arrange（前提条件）→ Act（操作）→ Assert（期待結果）の構造を読み取り、主要なケースを要約
3. **境界条件・エッジケース**: エラーケース、空配列、null/undefined、権限不足など、正常系以外のテストケースを列挙
4. **テストの種類**: Unit / DB integration / E2E のどれか（ファイル拡張子から判定）
5. **テストが使っているモック・フィクスチャ**: Factory、MockSet、テスト用のセットアップから、どの依存を差し替えているかを把握

### テスト↔実装のペアリング

テストファイルと対応する実装ファイルをペアにします:
- `foo.service.test.ts` → `foo.service.ts`
- `foo.model.test.ts` → `foo.model.ts`
- `foo.db-test.ts` → `foo.repository.ts` + 対応する UoW

ペアが見つかった場合は「テスト → 実装」の順で読む順序を推奨します。

---

## Step 4: ファイルの依存関係と読む順序の決定

変更されたファイルを分析して、推奨される読む順序を決定します。

### 優先度の判定ルール

以下の順序でファイルに優先度を付けます:

1. **最初に読む（テストで仕様を掴む）**
   - テストファイル（仕様書として読む — 「何をすべきか」を先に理解）
   - インターフェース・型定義・スキーマ（`.d.ts` / `interface` / `type` / `schema` / `.gql`）

2. **次に読む（コアロジックの確認）**
   - ビジネスロジック・サービス層（`services/` / `usecases/` / `domain/`）
   - データモデル・エンティティ（`models/` / `entities/`）
   - API エンドポイント・コントローラ（`controllers/` / `resolvers/`）

3. **最後に読む（詳細実装・周辺コード）**
   - リポジトリ実装・UoW（`infrastructure/`）
   - UI コンポーネント（`components/` / `views/` / `pages/`）
   - ヘルパー・ユーティリティ・設定ファイル

### 依存関係の検出

差分内の `import` / `require` / `use` 文を解析して、変更ファイル間の依存関係グラフを構築します。
依存される側（被 import 側）のファイルを先に読む順序を推奨します。

---

## Step 5: レビューガイドの出力

以下の形式で結果を出力します:

```
## 📋 変更概要 (Change Overview)

**変更カテゴリ**: [カテゴリ1] / [カテゴリ2] ...
**変更規模**: [Small / Medium / Large] （+XX行 / -XX行 / XX ファイル変更）

### このPR/変更で何が変わったか

[2〜5文で変更の目的と内容を説明。技術的な詳細より「なぜ・何を」を重視]

### 主な変更点

- [変更点1: ファイル名を含む具体的な説明]
- [変更点2]
- [変更点3]
（最大5つ）

### レビュアーが注意すべき点

- [破壊的変更・副作用・リスクがある箇所があれば記載]
- なければ「特になし」

---

## 🧪 テスト仕様ガイド (Test Spec Guide)

テストコードから読み取れる仕様です。実装コードを読む前にここで振る舞いを把握してください。

### [テストファイル名1] （Unit / DB / E2E）

**テスト対象**: `対応する実装ファイル名`

**この機能が満たすべき仕様:**

| # | 仕様（テストケースから抽出） | 区分 |
|---|---------------------------|------|
| 1 | [describe/it のラベルから抽出した振る舞い] | 正常系 |
| 2 | [振る舞い] | 正常系 |
| 3 | [エラーケースの振る舞い] | 異常系 |
| 4 | [境界条件] | エッジケース |

**前提条件・セットアップ:**
- [Factory/MockSet/フィクスチャで構築されているテストデータの概要]
- [モックされている依存: リポジトリ名、外部サービス名など]

**注目ポイント:**
- [テストから読み取れるビジネスルール・制約・不変条件があれば記載]

（テストファイルが複数ある場合、上記を各ファイルごとに繰り返す）

---

## 📖 推奨読書順 (Recommended Reading Order)

テストで仕様を掴んでから実装を読む順序を推奨します。

### 1. テストで仕様を掴む

| ファイル | 理由 |
|---------|------|
| `path/to/file.test.ts` | [このテストで把握できる仕様の概要] |

### 2. 型・インターフェースで構造を理解

| ファイル | 理由 |
|---------|------|
| `path/to/file.model.ts` | [理由] |
| `path/to/file.gql` | [理由] |

### 3. コアロジックの確認

| ファイル | 理由 |
|---------|------|
| `path/to/file.service.ts` | [理由] |

### 4. 周辺コード・詳細実装

| ファイル | 理由 |
|---------|------|
| `path/to/file.repository.ts` | [理由] |

---

## 💡 レビューのヒント

- **重点確認ポイント**: [この変更で特に注意して見るべきロジックや箇所]
- **テストカバレッジ**: [テストが追加/修正されているか、カバーされているか]
- **テストで未カバーの懸念**: [テストされていないが確認すべき振る舞いがあれば記載]
- **関連ファイル（変更なし）**: [変更はされていないが、影響を受ける可能性があるファイル]
```

---

## Notes

- ファイル数が多い（>15）場合は、読書順を「グループ」でまとめて表示する
- PR 指定時は PR のタイトル・説明文も考慮して概要を生成する
- 日本語で出力する（ファイル名・技術用語はそのまま英語）
- 変更内容が自明な場合（例: README のみ）は読書順セクションを省略してよい
- テスト仕様ガイドは変更差分にテストファイルが含まれる場合のみ出力する（テストがなければセクション自体を省略）
- テスト仕様の表では、仕様を多くても10件程度に絞り、細かいパラメータ違いは省略してよい
- テストファイルが多い（>5）場合は、関連するテストをグループ化して表示する
